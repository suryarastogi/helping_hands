## v325 — GitHub Issue Linking

(frontend FormState → SubmissionForm → useTaskManager → BuildRequest → Celery task →

- Comment posted on the issue linking to the created/updated PR
- Error handling: comment failures logged but don't block PR creation

**GitHubClient:** `get_issue()` and `create_issue_comment()` methods added.

## v326 — Create New Issue from Task

**Auto-create issue from task:** Added `GitHubClient.create_issue()` method for
creating issues via PyGithub API. New `create_issue` boolean field through the
full stack (frontend checkbox → BuildRequest → Celery task). When enabled and
no `issue_number` is manually provided, `_try_create_issue()` helper auto-creates
a GitHub issue with `[helping-hands]` prefixed title (first 120 chars of first
prompt line) and full prompt as body, applying the `helping-hands` label.

The created issue number flows into the existing `issue_number` pipeline — PR
gets "Closes #N" and a comment is posted on the issue linking to the PR.

**Error handling:** Issue creation failures are logged and reported in task
updates but do not block the build from proceeding.

**Frontend:** "Create issue" checkbox in Advanced settings, URL query param
support (`?create_issue=true`).

**Tests:** 5 new GitHubClient tests (create with title/body, with labels, null
body, empty title, whitespace title). 4 new Celery `_try_create_issue` tests
(success, title truncation, multiline prompt, error swallowing). 4 new frontend
tests (2 SubmissionForm checkbox, 2 useTaskManager submit body). 728 frontend
tests total (up from 724), 97 backend GitHub tests (up from 92).

## v327 — Sync Task Status with GitHub Issue

**Issue lifecycle sync:** When a task has a linked `issue_number`, GitHub issue
labels and comments are updated to reflect task progress:
- `_sync_issue_started()` adds `helping-hands:in-progress` label when the hand begins
- `_sync_issue_completed()` posts completion comment with PR link + runtime,
  swaps label to `helping-hands:completed`, removes `in-progress`
- `_sync_issue_failed()` posts failure comment with error details, swaps label
  to `helping-hands:failed`, removes `in-progress`

**GitHubClient:** `add_issue_labels()` (creates missing labels on repo) and
`remove_issue_label()` (silently ignores absent labels) methods added.

**Progress metadata:** `issue_number` included in Celery `_update_progress()`
metadata so frontend can track it during polling.

**Frontend:** `linkedIssueNumber` computed in `useTaskManager` hook from payload.
MonitorCard shows blue `#N` issue badge in header when a linked issue is present.
`taskInputs` includes `Issue: #N` item.

**Tests:** 7 new backend GitHubClient tests (3 add_issue_labels, 4 remove_issue_label).
8 new celery sync helper tests (2 _sync_issue_started, 3 _sync_issue_completed,
3 _sync_issue_failed). 4 new frontend tests (2 MonitorCard badge, 2 useTaskManager
linkedIssueNumber). 732 frontend tests total (up from 728), 104 backend GitHub
tests (up from 97).
